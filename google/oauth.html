煎餃 ※ 地端部署系列 ※ 02

n8n 接 Google 服務

Drive、Gmail、Google Sheets 都要過這關。流程是設一把 OAuth 鑰匙給 n8n 用，不是 API Key 那種一串字，而是要在 GCP 跟 n8n 之間來回接線兩次。

Drive Gmail Sheets OAuth 2.0 用戶端

等一下，為什麼這次不是 API Key？

因為你要 n8n 去動「你的」 Drive、Gmail、Sheets 資料。Google 不能光憑一串密碼就把你的東西交給 n8n，它要你「親自授權」給 n8n。授權這件事走的是另一套機制，叫做 OAuth 2.0。

差別像這樣：API Key 像是「房子的鑰匙」（給誰就能進去）；OAuth 像是「飯店的房卡」（要本人去櫃台辦理、授權給特定人、可以隨時撤銷）。Drive 跟 Gmail 這種會碰到你個人資料的，Google 強制要走 OAuth。

三方關係：你、n8n、Google

你
資料主人
Gmail / Drive / Sheets 都是你的東西，你說了算
仲介
n8n
想代表你去存取你的 Google 資料做自動化
資料保管者
Google
手上有你的資料，但只認你授權給誰

所以我們要做的事就是：到 Google 那邊登記一個叫 n8n 的「應用程式」（PART 1-3 在 GCP 做），然後在 n8n 那邊用這個身份去敲 Google 的門（PART 4 在 n8n 做）。中間 Google 會跳出視窗問你「你真的要授權 n8n 嗎？」按同意之後，n8n 才能用。

PART 1
在 GCP 啟用 Drive / Gmail / Sheets 三個 API
先讓 Google 知道你打算用這幾個服務。不啟用，後面就算授權也沒用。
一

登入 GCP Console，建一個叫 n8n 的 Project

GCP
為什麼要做這件事

GCP（Google Cloud Platform）用 Project（專案） 來分裝你的東西。一個 Project 像一個資料夾，裡面放著你啟用的 API、建立的用戶端、產生的金鑰。

我們建一個專門給 n8n 用的 Project，未來所有 n8n 相關的 Google 設定都集中在這裡，乾淨好管。

操作步驟
  1. 去 console.cloud.google.com，用你常用的 Google 帳號登入。
  2. 第一次登入的話：選國家 Taiwan、勾同意條款、按 Agree and Continue。
  3. 登入後，點 Console 頂端中央的 Project 下拉選單（顯示 Select a project 或預設專案名）。
  4. 跳出視窗右上角點 NEW PROJECT。
  5. Project name 填 n8n，按 CREATE，等 5-10 秒。
  6. 建好後，回頂端下拉選單，切換到剛建好的 n8n Project。確認頂端顯示 n8n 才能繼續。
前往 Google Cloud Console
第一次用 GCP 會要你綁信用卡。Google Drive / Gmail / Sheets 這幾個 API 完全免費，個人用根本不會扣錢，放心綁。
二

啟用 Drive、Gmail、Sheets 三個 API

GCP
為什麼要做這件事

GCP 預設什麼都沒開。Drive API、Gmail API、Sheets API 是三個獨立的功能開關，要逐一打開。

三個一次啟用完最省事，未來 n8n 隨時想用哪個都行，不用再回來 GCP 開東西。

操作步驟
  1. 在 Console 頂端的搜尋列，輸入 Google Drive API。
  2. 點搜尋結果裡的 Google Drive API（有 Google 圖示那個）。
  3. 進到 API 介紹頁，按藍色的 ENABLE 按鈕，等幾秒看到啟用完成。
  4. 重複上面三步，分別啟用 Gmail API 跟 Google Sheets API。
啟用是一次性動作。同一個 Project 下，一個 API 啟用過就永遠是開的，不用每次都重啟用。
PART 2
設定 OAuth 同意畫面
這是 Google 強制的：要建 OAuth 用戶端之前，先告訴 Google 你的 App 是誰、要做什麼、誰能用。不先設這個，第三段建用戶端的按鈕會是灰的，按不下去。
一

進 OAuth 同意畫面，選 User Type 為 External

GCP
為什麼要做這件事

OAuth 授權時 Google 會跳出一個視窗給使用者看，寫著「某某 App 要存取你的 Gmail，你同意嗎？」這個視窗的內容就是同意畫面（Consent Screen），要先設好它，才能建用戶端。

User Type 兩個選擇：Internal 只給同公司 Workspace 帳號用（要 Google Workspace 才能選），個人用戶都選 External。

操作步驟
  1. Console 左上角點選單（三條橫線），找 APIs & Services，子選單點 OAuth consent screen。
  2. 看到 User Type 選擇：External（個人用戶必選這個）。
  3. 按 CREATE。
二

填基本資料 + 加 Scopes（要授權哪些 API）

GCP
為什麼要做這件事

同意畫面上要顯示「這個 App 叫什麼名字、出問題找誰、要存取你哪些東西」。所以要填 App 名稱、聯絡 email、還有 scopes。

Scopes 就是權限範圍。你可以選「只讀 Drive」「能寫 Gmail」「能改 Sheets」之類。授權時 Google 會把這些列出來給使用者看：「你要授權 n8n 做以下幾件事」。

操作步驟（App information）
  1. App name 填 n8n。
  2. User support email 選你的 Gmail。
  3. App logo、App domain 全部留空（不需要）。
  4. Developer contact information 填你的 Gmail。
  5. 按 SAVE AND CONTINUE 進下一頁 Scopes。
操作步驟（Scopes）
  1. 按 ADD OR REMOVE SCOPES。
  2. 跳出視窗，在搜尋列分別搜尋並勾選：
    • https://www.googleapis.com/auth/drive（Drive 全權）
    • https://www.googleapis.com/auth/gmail.modify（Gmail 讀寫）
    • https://www.googleapis.com/auth/spreadsheets（Sheets 全權）
  4. 勾完按 UPDATE，回到 Scopes 頁面，按 SAVE AND CONTINUE。
Scopes 越寬，授權後能做的事越多，但風險也越高。先用上面這三個全權 scopes 練習。如果你只想讓 n8n 讀信不寫信，就改用 gmail.readonly，按需求調整。
三

加 Test Users（不加會無法授權）

GCP
為什麼要做這件事

剛建的 App 預設是「測試模式」（Testing），只有列在 Test users 清單裡的 Google 帳號才能授權。其他人按授權會被 Google 擋下來說「這個 App 還沒通過審查」。

所以一定要把你自己的 Gmail 加進去，否則待會在 n8n 按授權會失敗。

操作步驟
  1. 進到 Test users 頁面，按 + ADD USERS。
  2. 輸入你自己的 Gmail（就是你要被 n8n 代理存取的那個帳號）。
  3. 按 ADD，再按 SAVE AND CONTINUE。
  4. 最後一頁 Summary 直接按 BACK TO DASHBOARD。同意畫面就設好了。
個人用根本不用「發布」這個 App，永遠停在 Testing 狀態就好。把要用的帳號全加進 Test users，授權都能跑。Google 那段「未驗證 App」警告畫面，按進階繼續就好。
PART 3
建立 OAuth 用戶端 ID
同意畫面設好了，現在才能建用戶端。注意：這段先建一半，Redirect URI 留空。Redirect URI 要從 n8n 那邊拿，第四段再回來補。
一

進 Credentials 頁面

GCP
為什麼要做這件事

Credentials（憑證）是 GCP 管理所有用戶端跟 Key 的地方。所有授權相關的設定都在這裡建、這裡刪。

操作步驟
  1. Console 左上角選單，APIs & Services → Credentials。
  2. 頁面預設是空的（沒有任何用戶端或 Key）。
二

按「Create Credentials」建立 OAuth 用戶端 ID

GCP
為什麼要做這件事

Credentials 頁面可以建好幾種憑證（API Key、OAuth 用戶端 ID、服務帳號）。我們要的是 OAuth client ID，因為要做的是「使用者授權」流程。

操作步驟
  1. 頁面頂端按 + CREATE CREDENTIALS。
  2. 下拉選單選 OAuth client ID。
三

Application type 選「Web application」，命名為 n8n

GCP
為什麼要做這件事

Application type 有幾種選擇（Web、Desktop、Android、iOS、TV 等），給 n8n 用一定要選 Web application（網頁應用程式）。

為什麼？因為 n8n 是跑在瀏覽器裡的網頁應用。授權完成後，Google 要把你「導回」n8n 的網址，這個機制只有 Web application 才支援。

操作步驟
  1. Application type 選 Web application。
  2. Name 填 n8n。
  3. Authorized JavaScript origins 留空。
  4. Authorized redirect URIs 先留空（這裡的內容要從 n8n 那邊拿，下一段教你拿）。
  5. 按 CREATE。
  6. 跳出視窗顯示 Client ID 跟 Client secret，先別關。下一段要用。
邏輯銜接：現在你卡在「需要 Redirect URI 但還沒有」。這正常。Redirect URI 是 n8n 啟動時自動生成的網址，要去 n8n 那邊看才知道。接下來進 PART 4 去拿。
▷ 接線 PART 4 ◁
在 n8n 跟 GCP 之間來回接線
這段是最容易卡關的「邏輯交叉點」。簡單講：n8n 給 GCP 一個 URL（讓 Google 知道授權完要把人送回哪），GCP 給 n8n 兩串字（ID + Secret，讓 n8n 證明自己是合法的）。雙向都填完才會通。
為什麼要交叉做

OAuth 授權的核心是「雙方都要知道對方」。Google 要知道「授權完把使用者送回哪」（這就是 Redirect URI），n8n 要知道「我代表的是哪個 App」（這就是 Client ID + Secret）。

這兩串資訊一個在 n8n 手上、一個在 Google 手上，所以必須在兩邊之間貼來貼去。沒辦法只在一邊做完。

一

開 n8n，建一個 Google Drive OAuth2 憑證

n8n
為什麼要做這件事

n8n 那邊需要為每種 Google 服務建一份「Credential（憑證）」。我們先建 Google Drive 的，把 Redirect URL 拿出來，回 GCP 補上。

Gmail 跟 Sheets 之後也要建各自的憑證，但流程一樣，都會用到剛剛在 GCP 建的同一個 Client ID 跟 Secret。一次設定，三個共用。

操作步驟
  1. 打開 n8n（http://localhost:5678）。
  2. 左側選單點 Credentials，再按 + Add Credential。
  3. 搜尋 Google Drive OAuth2 API，點進去。
  4. 會看到一個欄位叫 OAuth Redirect URL，那是 n8n 自動產生的，長得像 http://localhost:5678/rest/oauth2-credential/callback。
  5. 點右邊的複製按鈕，把這個 URL 複製起來。
n8n 視窗先別關。等一下還要回來貼 Client ID 跟 Secret。直接開新分頁去 GCP 補資料就好。
二

回 GCP，把 Redirect URL 貼到剛建的用戶端

GCP
為什麼要做這件事

Google 收到 Redirect URL 之後，就知道「授權完成後要把使用者送回這個網址」。沒填這個，Google 會擋下授權說「Redirect URI mismatch」。

操作步驟
  1. 切回 GCP Console 的 Credentials 頁面。
  2. 找到剛建的 n8n 用戶端（OAuth 2.0 Client IDs 區塊），點進去。
  3. 找到 Authorized redirect URIs 區塊，按 + ADD URI。
  4. 把剛從 n8n 複製的 URL 貼上去（http://localhost:5678/rest/oauth2-credential/callback）。
  5. 按 SAVE。
  6. 回 Credentials 頁面，再點 n8n 用戶端，這次右上角會看到 Client ID 跟 Client secret，準備複製這兩串。
三

回 n8n，把 GCP 給的 Client ID + Secret 貼進去

n8n
為什麼要做這件事

Client ID 是 n8n 的「身份證號碼」，Client Secret 是它的「密碼」。Google 認得這兩串，才會把授權當作合法。

沒填這兩串，n8n 按 Sign in with Google 會直接失敗。

操作步驟
  1. 把 GCP 視窗的 Client ID 複製，貼到 n8n 那邊對應欄位。
  2. 同樣方式把 Client secret 複製、貼到 n8n。
  3. n8n 那個憑證頁面就填齊了：Redirect URL（自動）、Client ID（你貼的）、Client Secret（你貼的）。
Client Secret 跟密碼一樣，不要外流、不要 push 到 GitHub。萬一外流就回 GCP 重新產生。詳細安全守則在頁面最下方。
四

按「Sign in with Google」授權，看到 Connected 就成功

n8n
為什麼要做這件事

所有資料填好之後，按授權按鈕，n8n 會跳出 Google 的授權視窗（也就是 PART 2 設定的那個同意畫面）。在那個視窗按同意，n8n 才真正拿到「以你身份操作」的權限。

操作步驟
  1. 在 n8n 憑證頁面，按 Sign in with Google 按鈕。
  2. 會跳出新視窗讓你選 Google 帳號，選你剛在 Test users 加的那個帳號。
  3. 會看到一個警告「Google 尚未驗證此 App」，按 進階 → 前往 n8n（不安全）。這正常，因為這是你自己的 App，不會通過 Google 正式審查。
  4. 看到授權清單（要存取你的 Drive、Gmail、Sheets），按 繼續 或 允許。
  5. 視窗會自動關閉，回到 n8n。憑證頁面標題出現綠色 Connected 字樣，就成功了。
  6. 按 Save 把這個憑證存下來，給未來的 workflow 用。

Drive 憑證搞定了

Gmail 跟 Sheets 也要用？回 n8n Credentials 重複 PART 4 第一步，分別建 Gmail OAuth2 API 跟 Google Sheets OAuth2 API 憑證，同樣用剛剛 GCP 那組 Client ID + Secret。Redirect URL 都一樣，不用再回 GCP 改。

OAuth 用戶端安全守則

  • Client Secret 不要外流。外流的話，別人可以冒充你的 App 騙別人授權。發現外流立刻去 GCP 把 Client 刪掉，重建一個。
  • 不要 push 到 GitHub。n8n 的 credentials 存在 C:\n8n 那個 volume 裡，如果你不小心把 C:\n8n 整個推上去就完了。把它加進 .gitignore。
  • App 保持 Testing 狀態。個人用根本不需要發布，Testing 就能跑。發布反而會被 Google 抓去審查，要提交大量文件。
  • 授權範圍越小越好。如果你只是要讀 Gmail，scope 用 gmail.readonly 就好，不要選 gmail.modify。萬一外流災情會小很多。
  • 定期回 Google 帳號檢查。Google 帳號的「安全性」頁面有「具有帳戶存取權的第三方應用程式」，可以看哪些 App 還拿著你的授權。不用了的 App 撤銷授權。
OAuth Client Secret 跟 API Key 不一樣，外流的危害可能更大。因為 OAuth Client 可以冒充你的身份去騙別人授權他們的帳號。一旦外流請立刻處理。

下一步：開始用 Drive / Gmail / Sheets 做自動化

憑證有了之後，回 n8n 建 workflow 時拖一個 Google Drive / Gmail / Sheets 節點，在 Credential 欄選你剛建的這個憑證，就可以開始操作了。
